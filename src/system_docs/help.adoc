= Folio help: writing notes in AsciiDoc
:tags: help

Welcome to Folio. Every note you write here is plain text marked up in
*AsciiDoc* — a lightweight format where a few punctuation characters
stand in for bold, headings, lists, and so on, so the raw text stays
readable even before it is rendered. You do not need to know AsciiDoc
already: this reference teaches the small slice of it that Folio
understands, one construct at a time.

You edit a note in the *Source* view and read it in the *View* view; the
toggle at the top switches between them. The View view is exactly what
the examples below preview. Folio accepts a deliberately *strict subset*
of AsciiDoc, so most of what you write is ordinary prose with the
occasional marker. Two promises make that subset safe to live in:
well-formed markup renders, while a malformed inline marker surfaces a
clear _error_ in the View rather than quietly rendering garbage — and a
note still saves while you are mid-edit, so a half-finished line never
costs you your work.

This page is itself a Folio note, written in the very subset it
documents. The full AsciiDoc language is much larger; for anything beyond
this subset, the official documentation lives at
https://docs.asciidoctor.org/asciidoc/latest/[the AsciiDoc language docs]
and a one-page tour is the
https://docs.asciidoctor.org/asciidoc/latest/syntax-quick-reference/[syntax quick reference].

== Structure

Every construct below follows the same shape: first *how it renders*,
then *the source that produced it* in a code block, then the one limit
worth remembering. Here is a whole tiny note, end to end, to show how the
pieces fit together:

----
= Shopping
:tags: home, errands

A quick list before the *weekend*.

== To buy

* Milk
* Bread
----

That source is a *title* (`= Shopping` — a single `=`), a `:tags:` line that
files the note under two tags, a paragraph with one bold word, a
*section heading* (`== To buy`), and a short *list*. The rest of this
reference takes those pieces one at a time.

=== Title and tags

A note's first non-blank line may be its title, written `= Title` — one
`=`, a space, then the text. A title uses a *single* `=`, and only the
very first line may be one.

Directly under the title, an optional `:tags:` line files the note under
one or more comma-separated tags. Together they look like this:

----
= My note title
:tags: work, ideas
----

Limit: the title is valid *only* on the first line; a `=` line anywhere
later in the note is rejected rather than treated as a heading.

=== Section headings

Headings inside the body use two to six `=` signs — `==` through
`======`. A heading with more `=` signs nests under one with fewer; a
heading with equal-or-fewer `=` signs closes it and starts a new one.
(The official docs count these as "levels", where the title is level 0
and a `==` heading is level 1 — always one less than the number of `=`
signs. This page just counts the `=` signs, to sidestep that off-by-one.)

The subsection titles you are reading on this page — "Title and tags",
"Section headings", and the rest — are themselves live headings, each a
level deeper than "Structure" or "Blocks" above them. Their source is
just the heading line, for example:

----
== A top-level section
=== A nested subsection
====== The deepest allowed heading
----

Limit: the single-`=` title is valid only as the first non-blank line
(blank lines above it are fine). A `=` line anywhere later in the note is
rejected rather than treated as a heading.

== Text & emphasis

Inline markup styles a span of text *within a single line*. Each style
below is shown first as it renders, then with the literal markers you
type. (See also the official
https://docs.asciidoctor.org/asciidoc/latest/text/[text formatting] page.)

* *bold* — type `*bold*` (a pair of asterisks)
* _italic_ — type `_italic_` (a pair of underscores)
* [.line-through]#struck through# — type `[.line-through]#text#`
* [.underline]#underlined# — type `[.underline]#text#`
* `monospace` — wrap the span in single backtick characters

A monospace span is special: its contents are shown *verbatim* and never
re-parsed, which is exactly why every markup sample on this page can sit
inside one and survive intact.

Limit: a marker does not nest inside the same marker (no bold inside
bold), though different markers may nest, such as bold inside italic. An
unpaired marker on a line is reported as an error rather than rendered
literally.

=== Hard line breaks

A paragraph normally reflows — a plain source newline becomes a space,
so short lines join into one. To force a visible break instead, end a
line with a space and a single plus sign. Roses are red, +
topazes are blue, +
and this line break was made with a trailing plus. A plus sign anywhere
else — as in `a+b` — has no special meaning and stays literal.

== Blocks

Blocks are the larger structures stacked down the page: links, lists,
code, quotes, admonitions, tables, and images. Each subsection shows the
rendered result first, then its source.

=== Links

A link can be written four ways. Three are for everyday addresses:

* a *bare URL* such as https://example.com — Folio links it automatically;
* a *labelled URL*, `URL[label]`, showing the label in place of the address;
* the *`link:` macro*, `link:URL[label]`, which spells the link out explicitly.

The fourth is a *passthrough* macro, `link:++URL++[label]`, for an address
that is awkward to link — the `++ ... ++` around the URL keeps every
character literal. Reach for it when the address itself contains a markup
character (most often an underscore), which the three forms above would
otherwise read as formatting and cut the link short.

A label is optional for the bare and labelled forms but *required* by both
macro forms. Rendered, a labelled link and a passthrough link both look
like ordinary links: https://example.com[the Folio website] and
link:++https://en.wikipedia.org/wiki/Naive_set_theory++[naive set theory].

Here are all four forms, end to end:

----
Bare:        https://example.com
Labelled:    https://example.com[the Folio website]
Macro:       link:https://example.com[the Folio website]
Passthrough: link:++https://en.wikipedia.org/wiki/Naive_set_theory++[naive set theory]
----

Only the schemes `http`, `https`, and `mailto` become links. The bare and
labelled forms are lenient about anything else — text that does not begin
with one of those schemes is simply left as ordinary words rather than
linked. The `link:` macro is stricter: a disallowed scheme is reported as
an error instead of passed through.

The passthrough earns its place on real URLs: an address such as
`https://en.wikipedia.org/wiki/Naive_set_theory` breaks at its first
underscore in every other form — the link stops at `...Naive` and `_set_`
is read as italics. Wrapping it as `link:++...++[label]` makes Folio take
the address verbatim, underscores and all.

Limit: links never nest inside other links, and a label is mandatory for
either macro form.

=== Lists

An *unordered* list uses a leading `*`; an *ordered* list uses a leading
`.`. Rendered, they look like this:

* First point
* Second point
* Third point

. First step
. Second step
. Third step

----
* First point
* Second point
* Third point

. First step
. Second step
. Third step
----

Lists *nest* up to three levels deep: repeat the marker once more for
each level (`**` or `..` for level two, `***` or `...` for level three).
A space must follow the marker run. Levels may mix freely — an ordered
sublist can sit under an unordered item and the reverse:

* Produce
** Apples
** Pears
* Dairy
.. Milk
.. Cheese

----
* Produce
** Apples
** Pears
* Dairy
.. Milk
.. Cheese
----

Limit: nesting stops at three levels, and list items are
*single-line only* — no continuation lines. A list must start at the top
level (a single `*` or `.`), and each level may deepen by only one marker
at a time.

=== Code blocks

A code block is fenced by a line of four hyphens (`----`) above and below
it. An optional `[source,language]` directive may sit on the line before
the opening fence — `[source,python]` is accepted — but Folio does not yet
syntax-highlight the rendered View, so the language is recorded without
any visible effect for now. Its body is shown verbatim and is never
re-parsed, so any markers inside stay literal. Rendered:

[source,python]
----
def hello():
    print("Hello!")
----

The source is the same three-line body wrapped in `----` fences, preceded
by `[source,python]`. (See the official
https://docs.asciidoctor.org/asciidoc/latest/verbatim/source-blocks/[source blocks]
page.)

Limit: because the fence itself is four hyphens, a code block cannot
contain a line that is only four hyphens — that line would close it.

=== Quotes

A block quote is fenced by four underscores (`____`), with an optional
`[quote, Author, Source]` attribution line before it. Rendered:

[quote, Ada Lovelace, Notes]
____
That brain of mine is something more than merely mortal, as time will
show.
____

Its source pairs the attribution with the `____` fences:

----
[quote, Ada Lovelace, Notes]
____
That brain of mine is something more than merely mortal, as time will
show.
____
----

Limit: a quote's body is paragraphs only — no lists, tables, or other
blocks nested inside.

=== Admonitions

An admonition calls out a `NOTE`, `TIP`, `IMPORTANT`, `WARNING`, or
`CAUTION`. The single-line form is `KIND: text`, rendered like this:

NOTE: This is the single-line admonition form.

The block form pairs a `[KIND]` directive with a `====` fence (four equals
signs) around a body of one or more paragraphs (inline text only — no
nested blocks):

[TIP]
====
This is the block admonition form.
====

The source of both forms:

----
NOTE: This is the single-line admonition form.

[TIP]
====
This is the block admonition form.
====
----

Limit: only the five kinds above are recognised; the body of the block
form is inline text only, not nested blocks.

=== Tables

A table is fenced by `|===`, with one `|cell` per cell. The first row is
read as the header. Rendered:

|===
|Markup |Renders as
|*stars* |bold text
|_underscores_ |italic text
|===

Its source places each cell after a leading `|`:

----
|===
|Markup |Renders as
|*stars* |bold text
|_underscores_ |italic text
|===
----

Limit: each cell is *single-line and inline-only*, and cell text does not
wrap — a cell too wide for its column is truncated with an ellipsis
rather than flowing onto a second line. Keep cells short. (The full table
syntax is in the official
https://docs.asciidoctor.org/asciidoc/latest/tables/build-a-basic-table/[build a basic table]
page.)

=== Images

An image is placed with the `image::file[]` block macro. It renders
inline at its natural size, scaled down only if it would be wider than
the reading column:

image::help-demo.png[]

The source is the filename followed by a required pair of brackets:

----
image::help-demo.png[]
----

In full AsciiDoc those brackets carry attributes such as alt text and a
`width`, but Folio's subset does *not* interpret them: anything you place
between the brackets is ignored, and the image always renders at its
natural size capped at the column width.

Limit: you cannot set an image's size or alt text from the source —
sizing is automatic. The brackets must still be present and balanced,
with no further `[` or `]` between them.
